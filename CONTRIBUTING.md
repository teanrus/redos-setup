# Contributing

Спасибо за интерес к проекту `redos-setup`.

Этот репозиторий содержит shell-скрипт для автоматизации настройки рабочих мест на РЕД ОС. Основная цель изменений в проекте: сохранить предсказуемость сценария установки, прозрачность системных изменений и соответствие между кодом, релизными артефактами и документацией.

## Scope

Перед отправкой изменений важно учитывать:

- проект ориентирован на РЕД ОС `7.x` и `8+`;
- для РЕД ОС `7.x` поддерживается полный сценарий установки;
- для РЕД ОС `8+` компонент `ViPNet` должен рассматриваться отдельно, так как текущий релиз его не устанавливает;
- репозиторий использует GitHub Releases как источник устанавливаемых артефактов.

## Development Workflow

Рекомендуемый порядок работы:

1. Создайте отдельную ветку от актуального состояния `main`.
2. Внесите изменения в `scripts/setup.sh` и связанную документацию.
3. Проверьте, что логика для РЕД ОС `7.x` и `8+` остаётся согласованной.
4. Обновите `README.md`, если поведение скрипта изменилось.
5. Добавьте запись в `CHANGELOG.md`, если изменение влияет на пользователей.
6. Откройте pull request с кратким описанием мотивации и изменений.

## What To Test

Перед отправкой изменений желательно проверить:

- запуск скрипта на тестовой системе с РЕД ОС `7.x`;
- корректное определение РЕД ОС `8+` и автоматическое исключение несовместимых компонентов;
- отсутствие регрессий в интерактивных вопросах;
- корректность ссылок на артефакты GitHub Releases;
- соответствие текста в `README.md` реальному поведению скрипта.

Если полноценное тестирование недоступно, укажите это прямо в описании pull request.

## Style Notes

Для изменений в `scripts/setup.sh`:

- сохраняйте shell-логику простой и линейной;
- предпочитайте явные проверки вместо неочевидных сокращений;
- не добавляйте лишние зависимости без необходимости;
- используйте понятные сообщения для пользователя;
- учитывайте, что скрипт запускается с правами `root`.

Для документации:

- используйте нейтральный технический стиль;
- не описывайте поддержку того, что не проверено;
- синхронизируйте текст с фактической логикой скрипта.

## Pull Request Checklist

Перед открытием PR проверьте:

- изменение решает конкретную проблему;
- обновлены связанные документы;
- отражены ограничения по совместимости;
- в релизы не добавлены артефакты для неподдерживаемой версии ОС;
- в PR указано, что именно было протестировано.

## Release Notes

Если изменение затрагивает установку, совместимость, список компонентов или системные действия скрипта, добавьте краткую запись в `CHANGELOG.md`.
